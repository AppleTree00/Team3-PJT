# **📖 a4u (Apply4U) 프로젝트 공식 용어집 및 협업 사전**

**최종 업데이트**: 2026\. 06\. 22

**문서 목적**: PM, 기획, UI/UX, 백엔드/DB, QA 팀원(Human) 및 각 역할별 AI 에이전트 간의 커뮤니케이션 오해를 방지하고, 3일 완성 MVP의 제약사항과 기술적 합의를 동일한 수준으로 이해하기 위함입니다.

## **1\. 프로젝트 핵심 및 비즈니스 용어 (Business & Product)**

* **a4u (Apply4U)**  
  * **정의**: 본 프로젝트의 공식 서비스명. "인공지능으로 커리어의 가치를 높이세요"라는 슬로건을 가진 AI 기반 이력서 작성 및 커리어 코칭 플랫폼입니다.  
* **3일 완성 MVP (3-Day MVP)**  
  * **정의**: 3일이라는 극단적인 시간 제한 내에 시연(Demo)이 가능하도록 만든 최소 기능 제품.  
  * **팀 제약사항**: 모든 기획과 개발은 '확장성'보다 '시연의 완결성'에 집중합니다. 실제 구현되지 않은 기능은 과감히 '미구현 팝업'으로 대체합니다.  
* **시연용 3종 샘플 (Demo 3-Samples)**  
  * **정의**: MVP 시연을 위해 하드코딩된 3가지 페르소나 데이터.  
  * **구성**: 1\) IT 개발자형(김개발), 2\) 경영 관리자형(이매니저), 3\) 일반 범용형(박지원).  
  * **규칙**: 시스템은 이 3가지 데이터에 최적화되어 있으며, 이외의 데이터 입력 시 예외 처리를 우선합니다.  
* **50+ 시니어 타겟 (50+ Senior Persona)**  
  * **정의**: 1차 기관 시연의 주 타겟층.  
  * **디자인/UX 규칙**: 글씨 크기 확대, 직관적인 버튼(아이콘+텍스트), 친절한 대화형 안내(단일 대화 뷰)가 필수 적용되어야 합니다.

## **2\. AI 및 프롬프트 엔지니어링 용어 (AI & Prompt)**

* **AI 에이전트 (AI Agent)**  
  * **정의**: 각 팀원(PM, Dev, UI, QA)을 1:1로 보조하는 LLM (GPT-4o, Claude 등).  
  * **규칙**: 에이전트는 본인에게 할당된 역할(Role)의 .md 파일 지침만 수행하며, 타 도메인의 코드 수정은 제안만 할 뿐 직접 병합하지 않습니다.  
* **하이브리드 AI 폴백 (Hybrid AI Fallback)**  
  * **정의**: AI 코칭 요청 시의 다중 대응 로직.  
  * **흐름**: OpenAI(gpt-4o-mini) 호출 실패 시 ➔ Gemini(gemini-flash 계열) 호출 ➔ 모두 실패 또는 타임아웃 시 ➔ Mock(미리 준비된 시연용 텍스트) 반환.  
* **퓨샷 프롬프팅 (Few-shot Prompting)**  
  * **정의**: AI 코칭 퀄리티를 높이기 위해 백엔드에서 프롬프트를 전송할 때, 미리 잘 작성된 이력서 샘플 2\~3개를 함께 던져주어 답변의 형식을 강제하는 기법.

## **3\. 프론트엔드 및 UI/UX 용어 (Frontend & UI)**

* **Tailwind CSS (테일윈드)**  
  * **정의**: MVP의 반응형 웹 구현을 위해 사용하는 Utility-first CSS 프레임워크.  
  * **규칙**: 별도의 .css 파일을 최소화하고 CDN 방식으로 HTML 내부에 클래스를 직접 작성하여 속도를 높입니다.  
* **미구현 기능 방어 팝업 (handleUnavailableFeature)**  
  * **정의**: 3일 안에 개발하지 못하는 기능(구글 로그인, PDF 다운로드 등)을 클릭했을 때 호출하는 글로벌 JavaScript 함수.  
  * **출력 메시지**: *"현재 이 기능은 고도화 단계에 있습니다. 업데이트 이후 사용 가능하니 잠시만 기다려주세요."*  
* **단일 대화 뷰 (Single Chat View)**  
  * **정의**: 사용자가 복잡한 폼을 채우는 대신, 챗봇과 대화하듯 한 화면에서 이력서를 완성해 나가는 시니어 친화적 UI 패턴.  
* **로컬 상대 경로 (Local Relative Path)**  
  * **정의**: GitHub Pages 배포 시 이미지 엑스박스(Broken Image) 방지를 위해 사용하는 이미지 경로 방식. (예: /assets/img.png ➔ ./assets/img.png)

## **4\. 백엔드 및 데이터베이스 용어 (Backend & DB)**

* **Flask (플라스크)**  
  * **정의**: 백엔드 API 서버 구축을 위해 사용하는 파이썬 경량 웹 프레임워크. MVP의 빠른 라우팅(Routing) 구현을 위해 채택.  
* **SQLite 및 a4u.db**  
  * **정의**: 로컬 파일 기반의 경량 데이터베이스. 서버 설정 시간을 줄이기 위해 채택. github에는 푸시되지 않도록 .gitignore에 등록되어 있습니다.  
* **extra\_json (JSONB 패턴)**  
  * **정의**: models.py 데이터베이스 스키마에서 사용하는 확장 필드.  
  * **용도**: 사용자의 이력이나 기술 스택이 고정된 컬럼에 맞지 않을 때, JSON 형태로 자유롭게 데이터를 밀어 넣기 위한 유연한 필드입니다.  
* **HWPX 포맷 렌더링 전환 (HWPX Handling)**  
  * **정의**: 시니어들이 자주 올리는 '한글(HWP/HWPX)' 파일 처리 정책.  
  * **규칙**: 서버에서 무겁게 파싱하지 않고, 모달창을 띄워 \*"PDF나 WORD로 변환해서 올려주세요"\*라고 친절하게 우회(Bypass) 유도합니다.

## **5\. QA 및 협업 프로세스 용어 (QA & Collaboration)**

* **MD 파일 주도 개발 (MD-Driven Development)**  
  * **정의**: 코드 작성 전, 각 역할별 마크다운 파일(pm.md, UI.md, dev.md, QA.md)에 작업 내용을 먼저 명세하고 합의하는 방식.  
* **파일 분절 전략 (File Segmentation & Monopoly)**  
  * **정의**: Git 병합 충돌(Merge Conflict)을 막기 위해 팀원과 에이전트에게 파일 단위 독점권을 부여하는 규칙.  
  * **예시**: UI 파트는 .html만, 백엔드 파트는 \_routes.py와 models.py만 수정 권한을 가집니다.  
* **조건부 통과 (Conditional Pass)**  
  * **정의**: QA 테스트 중 결함은 발견되었으나, MVP 시연에 치명적이지 않아 '수용 가능한 수준'으로 판단하여 테스트를 통과시키는 상태.  
* **결함 심각도 (Defect Severity)**  
  * **Critical (치명적)**: 시스템 크래시, 파일 업로드 시 멈춤 현상 등 즉각 수정 대상.  
  * **Major (주요)**: 기능은 동작하나 UI가 깨지거나 브라우저 기본 Alert이 뜨는 등 UX 저하 요소.  
  * **Minor (경미)**: 오타, 약간의 여백 불일치 등 시연에 무리 없는 요소.

## **6\. GitHub 배포 용어 (Deployment)**

* **GitHub Pages (깃허브 페이지)**  
  * **정의**: 완성된 HTML/CSS/JS 정적 파일을 무료로 웹에 호스팅하여 클라이언트(기관)에게 시연 링크를 제공하기 위한 서비스.  
* **환경변수 (.env / .env.example)**  
  * **정의**: AI API Key 등 민감한 정보가 담긴 파일.  
  * **규칙**: .env는 절대 GitHub에 올리지 않으며(보안 필수), 팀원 간 설정 공유를 위해 빈 값으로 둔 .env.example만 깃에 공유합니다.

**\[팀원 행동 강령\]** \> *우리(휴먼)와 에이전트(AI)는 위 용어를 바탕으로 대화합니다. 확신이 서지 않는 구현 방식이 있다면 언제든 PM의 PROJECT\_MASTER.md를 확인하거나 슬랙/이슈 트래커에 이 용어집을 기반으로 질문해 주세요.*
